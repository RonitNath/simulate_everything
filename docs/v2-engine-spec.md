# simulate_everything V2 Engine Spec

## Context

This spec comes from a design session exploring how to make the game more interesting for AI-vs-AI spectating. The current engine (V1) is a turn-based territory-control game with integer armies on cells and fog of war. V2 is a ground-up engine rewrite that keeps the grid substrate but changes nearly everything else.

### Design goals
- **Hexagonal grid** for uniform distance, natural encirclement, and clean tactical formations
- Continuous-time simulation, not turn-based
- Units as discrete entities (companies), not anonymous integer counts on cells
- Terrain generated procedurally with varying economic value
- Engagement-based combat that locks units and punishes careless engagement
- Economy: units on resource tiles generate income, income buys new units
- Large, expansive maps with 5-tile vision and fog of war
- Foundation for future layers: unit types (cavalry, archers, siege), supply logistics, fortifications, technology, commanders

### What V2 is NOT (yet)
- No unit type variety (all units identical for now)
- No supply/transport logistics (resources teleport to general)
- No fortifications or structures
- No technology tree
- No morale or social mechanics
- No commanders or delegation

These are future layers. V2 is the foundation they build on.

## Hex Grid

### Why hex

Square grids have fundamental problems for tactical games:
- **4-direction**: encirclement requires only 4 units (too easy), diagonal gaps allow slipping through surrounds
- **8-direction**: diagonal moves cover √2 distance for the same cost, creating exploitable strategic artifacts. Every competent AI will move diagonally because it's strictly better. Engagement range is non-uniform.
- **Hex (6-direction)**: all six neighbors are equidistant. No distortion. Encirclement requires 6 units — a real commitment. Formations are natural. This is why every hex-and-counter wargame since the 1960s uses hex.

Hex rings scale uniformly: ring 1 = 6 hexes, ring 2 = 12, ring 3 = 18. This makes siege/encirclement scenarios (like the Battle of Alesia — concentric rings of besiegers and relief forces) geometrically natural.

### Coordinate system

Use **axial coordinates** (q, r) internally. Reference: [Red Blob Games hex guide](https://www.redblobgames.com/grids/hexagons/).

```
Axial coordinates (q, r):
- q increases to the right
- r increases downward-right (for pointy-top) or downward (for flat-top)
- The implicit third coordinate s = -q - r (cube constraint: q + r + s = 0)
```

**Recommended orientation**: flat-top hexes. This produces rows that align horizontally, making ASCII representation and frontend rendering more natural.

**Storage**: row-major in a `Vec<Cell>` using offset coordinates for indexing. Convert between offset and axial as needed for neighbor calculations and distance.

```rust
// Offset coordinates for storage (even-r offset for flat-top)
fn index(row: usize, col: usize, width: usize) -> usize {
    row * width + col
}

// The 6 hex neighbors (flat-top, even-r offset)
// For even rows: NE=(r-1,c), E=(r,c+1), SE=(r+1,c), SW=(r+1,c-1), W=(r,c-1), NW=(r-1,c-1)
// For odd rows:  NE=(r-1,c+1), E=(r,c+1), SE=(r+1,c+1), SW=(r+1,c), W=(r,c-1), NW=(r-1,c)
```

### Distance

Hex distance between two cells = `(|dq| + |dr| + |ds|) / 2` in cube coordinates, which equals the minimum number of hex steps between them.

### Implications for existing mechanics

- **Vision radius 5**: all hexes within hex distance 5. The implementation currently uses broader vision to support economy and convoy management.
- **Adjacency**: 6 neighbors, not 4 or 8
- **Engagement**: requires hex-adjacency (sharing a hex edge)
- **Pathfinding**: BFS/A* over 6 neighbors instead of 4
- **Movement**: units move to one of 6 adjacent hexes per move step

## Data Model

### Cell

```rust
struct Cell {
    terrain_value: f32,  // 0.0-3.0, production per tick when a unit occupies this cell
}
```

No `Tile` enum. No `owner` per cell. No mountains, no cities. All terrain is traversable. Territory is implicit — it's wherever your units are.

Terrain value is generated by layered Perlin noise (see Map Generation below).

### Unit

```rust
struct Engagement {
    enemy_id: u32,
    edge: u8,  // 0-5, which hex edge this engagement is on
}

struct Unit {
    id: u32,
    owner: u8,
    pos: Axial,              // axial coordinate
    strength: f32,           // 100.0 = full company, degrades in combat, <=0 = dead
    move_cooldown: u8,       // ticks remaining until this unit can move again
    engagements: Vec<Engagement>,  // active engagements, one per hex edge (max 6)
    destination: Option<Axial>,  // pathfinding target, unit moves toward this automatically
    is_general: bool,        // true for the player's general unit
}
```

Units are the fundamental entity. Each unit is a "company" — a formation of soldiers. Strength represents combat effectiveness (starts at 100, degrades through combat, unit dies at 0). A unit with any engagements is pinned and cannot move.

The general is a unit with a special flag (or a separate field on the player). Losing your general = losing the game.

### Player

```rust
struct Player {
    id: u8,
    resources: f32,      // accumulates from units on resource tiles
    general_id: u32,     // which unit is this player's general
    alive: bool,
}
```

### GameState

```rust
struct GameState {
    width: usize,            // grid width in offset coordinates
    height: usize,           // grid height in offset coordinates
    grid: Vec<Cell>,         // row-major using offset coordinates
    units: Vec<Unit>,        // all units in the game
    players: Vec<Player>,
    tick: u64,               // current simulation tick
    next_unit_id: u32,       // monotonically increasing counter for assigning unique unit IDs
}

// TICKS_PER_SECOND is a module-level constant (not a field on GameState)
```

The grid is stored using offset coordinates for cache-friendly access. Hex math (neighbors, distance, pathfinding) uses axial/cube coordinates, converting as needed.

## Map Generation

### Terrain generation via Perlin noise

Use layered Perlin noise (2-3 octaves) to generate a continuous terrain value map. The output is a float per cell in range [0.0, 3.0] representing economic productivity:

- **0.0-0.5**: Barren/wasteland (desert, rocky ground). Low production.
- **0.5-1.5**: Grassland/plains. Moderate production.
- **1.5-2.5**: Fertile land (river valleys, good soil). High production.
- **2.5-3.0**: Rich deposits (mines, extremely fertile). Very high production.

The noise should produce organic-looking terrain with large regions of similar value, not salt-and-pepper randomness. Use parameters:
- Octave 1: large features (scale ~0.03-0.05), high amplitude
- Octave 2: medium detail (scale ~0.1), medium amplitude  
- Octave 3: fine variation (scale ~0.3), low amplitude

Normalize the combined output to [0.0, 3.0].

**Hex-aware noise sampling**: when sampling noise for hex cells, convert offset coordinates to world-space positions that account for hex geometry (staggered rows, hex width vs height ratio). This prevents the noise pattern from being distorted by the offset grid.

Terrain also affects movement speed slightly: higher terrain value = slightly higher move cooldown (lush terrain is harder to cross). Exact formula TBD during implementation, but something like `base_cooldown + floor(terrain_value * 0.5)`.

### Balanced initial placement

After generating terrain, compute a "strategic value" heatmap: for each cell, sum the terrain values within a hex radius (e.g., 10 hexes). This gives a smooth map of which areas are economically valuable.

Place generals such that:
1. Each general is in a region with approximately equal strategic value (within some tolerance)
2. Generals are far apart (minimum hex distance based on map size)
3. Generals are not in the worst terrain (minimum terrain value at spawn point, e.g., > 1.0)

The placement algorithm:
1. Compute the strategic value heatmap
2. Find candidate positions meeting the margin and minimum-value constraints
3. For 2 players: find the pair of candidates that maximizes hex distance while minimizing strategic value difference
4. For N players: extend to N-way balanced placement

Each general starts with a small number of units nearby (e.g., 5 units within 3 hexes of the general).

### Map size

V2 targets larger maps than V1. Recommended sizes (in offset grid dimensions):
- Small: 60x60 (~3,600 hexes) — fast games, good for testing
- Medium: 100x100 (~10,000 hexes) — standard
- Large: 200x200 (~40,000 hexes) — epic games, tests scalability

The engine should work at any size. The frontend and observation model need to handle large maps gracefully (zoom levels, region summaries — but that's a frontend concern, not engine).

## Simulation Loop

### Continuous ticks, not turns

The simulation runs as a loop of discrete ticks. Each tick:

1. **Resource generation**: for each unit that is stationary (not moving, not engaged), add `cell.terrain_value * RESOURCE_RATE` to the unit's owner's resource pool.

2. **Unit production**: if a player has >= 10 resources and their general is alive and has an empty adjacent hex, deduct 10 resources and spawn a new unit on that hex with strength 100.0. (Auto-production or agent-triggered — TBD, probably agent-triggered via directive.)

3. **Combat resolution**: for each unit with engagements, compute effectiveness = `1/sqrt(N)` where N = number of engaged edges. For each engagement, apply damage: `damage_to_unit += enemy.strength * DAMAGE_RATE`, `damage_to_enemy += unit.strength * DAMAGE_RATE * effectiveness`. When a unit hits strength <= 0, it dies and all its engagements are cleared.

4. **Movement**: for each unit with a destination and move_cooldown == 0:
   - Gather one-step movement intents first; do not mutate positions immediately
   - Resolve same-target conflicts deterministically
   - Block movement into occupied hexes unless future explicit displacement rules say otherwise
   - Auto-create engagement for obvious movement contact cases: enemy swap attempts, enemy contests over the same destination, and blocked advances into an occupied enemy hex
   - Apply a light movement-facing zone-of-control rule so units already adjacent to an enemy cannot slide deeper through another enemy-adjacent hex unless the move increases distance from every current threat
   - Commit only the winning moves, then set move_cooldown from terrain and clear destination on arrival
   
5. **Cooldown decrement**: all units with move_cooldown > 0 decrement by 1.

6. **Engagement checks**: (agent-triggered via directive — see Engagement below)

7. **Death/cleanup**: remove dead units. Check if any player's general is dead → that player is eliminated. Check win condition.

8. **Tick increment**: tick += 1.

### Timing

The tick rate is configurable (default: 10 ticks per simulated second). The web server controls how fast ticks are dispatched (real-time, accelerated, or as-fast-as-possible for simulation mode).

## Combat: Edge-Based Engagement Model

Each hex has 6 edges (numbered 0-5, e.g., NE=0, E=1, SE=2, SW=3, W=4, NW=5). Engagements are tracked per-edge, meaning a unit can fight on multiple edges simultaneously — facing different enemies in different directions.

### Data model

```rust
struct Engagement {
    enemy_id: u32,
    edge: u8,  // 0-5, which hex edge this engagement is on
}

// In Unit:
engagements: Vec<Engagement>,  // up to 6, one per edge
```

A unit with `engagements.len() == 0` is free to move. A unit with any engagements is pinned (cannot move, destination cleared). This is the fundamental rule: **any engagement locks the unit in place.**

### Initiating engagement

Engagement is initiated by an agent directive: "unit X engages unit Y." Constraints:
- X and Y must be hex-adjacent (sharing a hex edge)
- The shared edge must be free on BOTH units (neither already engaged on that edge)
- X and Y must belong to different players

When engagement begins:
- Both units get an `Engagement` entry for the shared edge
- Both units stop moving (destination cleared, can't be set while any engagement exists)
- Combat damage begins applying each tick

Either adjacent unit can initiate engagement via explicit agent directive. This prevents passive camping from being risk-free — if you park next to an enemy, they can engage you on their terms.

### Damage during engagement

Each tick, damage is computed per-engagement and modified by how many edges the unit is fighting on:

```
For each unit U with N engagements:
    effectiveness = 1.0 / sqrt(N)   // fighting on more edges = spread thinner
    
    For each engagement with enemy E:
        damage_to_U += E.strength * DAMAGE_RATE
        damage_to_E += U.strength * DAMAGE_RATE * effectiveness
```

The `1/sqrt(N)` effectiveness curve means:
- **1 edge**: 100% effectiveness (full strength applied to that fight)
- **2 edges**: 71% effectiveness per fight (fighting on two fronts)
- **3 edges**: 58% effectiveness (stretched thin)
- **6 edges**: 41% effectiveness (completely surrounded, barely fighting back)

This creates powerful incentives:
- **Flanking**: engaging an already-engaged unit from a second edge is devastating. The defender's effectiveness drops from 100% to 71% across BOTH fights, while each attacker fights at 100%.
- **Alesia outer defense**: a unit engaged on edge 2 (facing besieged) AND edge 5 (facing relief army) fights at 71% on both fronts. It's viable but costly — the Alesia defenders were stretched thin, historically accurate.
- **Encirclement**: surrounding a unit on 4+ edges makes it nearly helpless. It deals 50% or less damage per fight while taking full damage from each attacker.
- **Concentration of force**: even without more total strength, engaging on more edges wins. Two weak units on two edges beat one strong unit because the strong unit's effectiveness is split.

`DAMAGE_RATE` is a tuning constant (start with 0.01 — meaning a full-strength unit on 1 edge deals 1.0 damage per tick to its opponent, so two equal 1v1 units each lose 1% strength per tick, taking ~50 ticks to resolve).

### Disengagement

An agent can order a unit to disengage from a **specific edge** (dropping one engagement) or **all edges** (full retreat). Cost per disengagement: **30% of current strength** is lost immediately.

Rules:
- Disengaging from one edge frees that edge on both units. If the unit has other engagements, it remains pinned.
- Disengaging from all edges at once still costs 30% total (not 30% per edge — one brutal rip).
- The opponent(s) become free on their respective edges (they can pursue, re-engage, or turn to face another threat).

Disengagement is a serious commitment:
- Don't engage unless you think you'll win
- If you're losing on one front, disengaging from that edge costs half your remaining strength — you might survive but you're crippled
- The winning side wants to stay engaged (opponent is weakening)
- The losing side faces a hard choice: stay and die, or disengage and limp away at half strength

**Surrounded units (engaged on 3+ edges) cannot disengage.** They are locked until they die or all attackers disengage/die. This makes encirclement decisive — once you close the ring, the trapped force is doomed. Breaking the ring from outside (the relief force killing one of the besieging units) is the only escape.

### Death

When a unit's strength reaches 0 or below, it is removed from the game. All its engagements are cleared — opponents on those edges become free. If the dead unit is a player's general, that player is eliminated and all their units are removed.

### No engagement = no combat

Units still cannot damage each other without engaging, and simple adjacency does NOT automatically start combat. The main tactical API remains explicit `Engage` / `Disengage`. The engine now adds a narrow movement-contact exception: some blocked movement conflicts auto-create engagement so enemies do not overlap or pass through each other unrealistically. In ordinary positioning, the agent must still order engagement. This means:
- Scouts can move past enemies without fighting (if the enemy doesn't engage them)
- Either side can initiate engagement when adjacent
- Positioning matters — you want to be adjacent on YOUR terms, not theirs
- You can stack units next to an enemy and engage from multiple edges simultaneously in a coordinated attack

### Tactical implications

The edge-based model creates rich tactical scenarios from simple rules:

**Battle line**: a row of units facing a row of enemies. Each unit is engaged on 1 edge (100% effectiveness). A clean, attritional fight. Whoever has more strength or more units wins slowly.

**Flanking**: send a unit around the line to engage an enemy on a second edge. That enemy drops to 71% effectiveness on BOTH fights. The flanked unit crumbles faster, creating a gap in the line.

**Pincer**: engage from two opposite edges (e.g., edges 1 and 4). The defender is fighting two directions at 71% each. Each attacker fights at 100%. Even if the defender is stronger, the split attention is devastating.

**Alesia / double siege**: your unit on edge 2 engages the besieged force. An enemy relief unit on edge 5 engages you. You're fighting on two fronts at 71%. You can hold if your strength is high enough, but you need reinforcement or you'll be ground down.

**Pocket / encirclement**: surround a unit on 4+ edges. It fights at 50% or less per edge. With 4 attackers each at 100%, the surrounded unit takes 4x the damage it deals. Encirclement is decisive — the winning move is to close the ring.

**Escape corridor**: when surrounding, deliberately leaving 1-2 edges open. The trapped unit can disengage (30% strength loss) and retreat through the gap. This is strategically useful when you want to destroy the unit's combat effectiveness without committing to a full encirclement (which requires 6 units).

## Vision and Fog of War

### Vision radius

Every unit has a vision radius of 5 hexes (hex distance ≤ 5). This covers the unit's hex plus 5 rings: 1 + 6 + 12 + 18 + 24 + 30 = **91 hexes** per unit.

The union of all a player's units' vision is that player's visible area. Everything outside is fog of war.

### What's visible in fog

- **Terrain**: always visible everywhere (terrain doesn't change, and you can see the landscape). The terrain value map is fully known to all players.
- **Units**: only visible if in a player's vision. Fog hides enemy units.
- **Resources**: player's own resource count is always known. Enemy resources are hidden.

### Spectator view

Spectators see everything (no fog). The spectator view should show both players' full state, including a visual indicator of each player's vision boundary (e.g., a colored outline around the perimeter of the visible hex region for each player).

## Agent Interface

### Observation

The agent receives observations periodically (not every tick — the polling rate is configurable or budget-limited). An observation includes:

```rust
struct Observation {
    tick: u64,
    player: u8,
    
    // Full terrain map (static, could be sent once)
    terrain: Vec<f32>,  // terrain_value per cell, row-major offset coords
    width: usize,
    height: usize,
    
    // Player state
    resources: f32,
    
    // Own units (full info)
    own_units: Vec<UnitInfo>,
    
    // Visible enemy units (only those in vision)
    visible_enemies: Vec<UnitInfo>,
    
    // Visibility mask (which cells the player can see)
    visible: Vec<bool>,
}

struct UnitInfo {
    id: u32,
    owner: u8,
    q: i32,              // axial hex coordinate
    r: i32,              // axial hex coordinate
    strength: f32,
    engagements: Vec<Engagement>,  // per-edge engagements (edge + enemy_id)
    is_general: bool,
}
```

### Directives

The agent issues directives (not per-unit-per-tick orders):

```rust
enum Directive {
    /// Move unit to a destination hex. Unit pathfinds automatically.
    Move { unit_id: u32, q: i32, r: i32 },
    
    /// Engage an adjacent enemy unit on the shared hex edge.
    /// The edge is computed automatically from the two units' positions.
    Engage { unit_id: u32, target_id: u32 },
    
    /// Disengage from a specific edge (frees that edge on both units).
    /// Costs 30% of current strength.
    DisengageEdge { unit_id: u32, edge: u8 },
    
    /// Disengage from ALL edges at once (full retreat).
    /// Costs 30% of current strength (not per-edge).
    DisengageAll { unit_id: u32 },
    
    /// Produce a new unit at the general's location (costs 10 resources).
    Produce,
    
    /// No action.
    Pass,
}
```

An agent can issue multiple directives per observation cycle. Invalid directives are silently ignored (e.g., engaging a non-adjacent unit, engaging on an occupied edge, producing with insufficient resources).

### Agent polling

In the round-robin/live modes, agents are polled for directives every N ticks (e.g., every 5 ticks = twice per simulated second at 10 ticks/sec). Between polls, units execute their existing orders autonomously (continue moving toward destination, continue fighting if engaged).

For future compute-budget mechanics: the agent could choose how many observations to read (costing budget) and how many directives to issue. But for V2 launch, simple periodic polling is fine.

## ASCII Representation

### Hex ASCII layout

Hex grids are represented in ASCII with staggered rows (odd rows indented by half a cell width):

### Terrain layer (static)
```
Terrain (0-3 scale, single digit)
   0  1  2  3  4  5  6  7
 0  1  1  2  2  3  2  1  1
  1  1  2  2  3  3  2  1  0
 2  0  1  2  2  2  1  1  1
  3  1  1  1  2  2  1  0  0
 4  0  0  1  1  1  1  1  1
```

Odd rows are indented by 2-3 characters to indicate hex offset. Each cell has 6 neighbors: for row 1, col 2 (value=2), the neighbors are row 0 cols 2,3 (NE/NW), row 1 cols 1,3 (W/E), row 2 cols 2,3 (SE/SW).

### Game state
```
Tick 1500 | a: 450 res, 24 units | b: 380 res, 19 units
    0    1    2    3    4    5    6    7
 0 .... .... a 85 .... .... .... .... ....
  1 .... A100 a 92 a 45 .... .... .... ....
 2 .... a 78 .... .... b 60 .... .... ....
  3 .... .... .... a*50 b*67 .... .... ....
 4 .... .... .... .... .... b 90 B100 ....
```

Encoding:
- `a 85` — player a's unit, strength 85
- `A100` — player a's general, strength 100
- `a*50` — player a's unit, engaged in combat, strength 50
- `....` — empty cell
- Capital letter = general

Multiple units on the same hex: show the strongest. (Crowding limits are a future mechanic — for V2 launch, units can share hexes but only one is shown in ASCII.)

## Frontend Rendering

### Hex rendering

Each hex cell is rendered as a flat-top hexagon. Use CSS `clip-path` or SVG polygons. Layout uses offset coordinates with every other row shifted right by half a hex width.

Hex dimensions: for a flat-top hex with width `w`, the height is `w * √3/2`. The horizontal spacing is `w * 3/4` (hexes overlap by 1/4 width). The vertical spacing is `w * √3/2`.

### Terrain
Each hex's background color reflects its terrain value:
- 0.0: dark brown/sandy (`#2a2218`)
- 1.0: muted green (`#1e3a1e`)
- 2.0: rich green (`#2a5a2a`)
- 3.0: bright green/gold (`#4a7a2a`)

Smooth gradient between these values. No discrete terrain types — the Perlin noise produces a continuous field and the colors reflect it.

### Units
Rendered as small circles or icons centered on their hex, colored by owner (existing player color palette). Size or brightness reflects strength (a strength-50 unit is dimmer/smaller than strength-100).

Generals get the existing gold border treatment.

Engaged units could have a red border or a connecting line between the two engaged units across the hex edge.

### Vision
Each player's visible area is indicated by a colored outline tracing the perimeter of their visible hex region. For player 0 (blue), a thin blue line around all hexes visible to blue. For player 1 (red), a thin red line. Spectators see both outlines.

Hexes outside both players' vision are slightly darker (fully fogged). Hexes visible to one player but not the other show only that player's outline.

## Open Questions

1. **Crowding/stacking**: Normal movement no longer allows same-hex military stacking. If future layers want crowding again, it should be a deliberate mechanic (for example garrisons, transports, or siege layering), not a default movement outcome.

2. **Auto-production vs agent-triggered**: should unit production happen automatically when resources >= 10, or only when the agent issues a `Produce` directive? Recommendation: agent-triggered (more strategic control).

3. **General combat**: can the general fight? Can it be engaged? If yes, it's a high-value target that needs protection. If no, it's invulnerable until the army is defeated. Recommendation: yes, the general can fight and be killed.

4. **Unit pathfinding**: BFS (simple, always finds shortest path on hex) vs A* (faster for large maps). Start with BFS, optimize if needed.

5. **Spectator tick rate**: in the web viewer, how fast should ticks play? The current game runs at configurable speed (25-500ms per turn). V2 could do similar — configurable ms per tick for spectating.

6. **Hex orientation**: flat-top vs pointy-top? Flat-top produces horizontal rows (natural for ASCII). Pointy-top produces vertical columns. Recommendation: flat-top for ASCII readability.

### Resolved decisions

- **Multiple engagements**: YES. Units can be engaged on multiple hex edges simultaneously (up to 6). Effectiveness scales as `1/sqrt(N)` where N = number of engaged edges. Units engaged on 3+ edges cannot disengage (surrounded). This is a V2-launch feature, not deferred — it's core to the tactical model.

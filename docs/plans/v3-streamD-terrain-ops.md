# Stream D: Terrain Operation Log

Status: **ready for implementation**
Depends on: Phase 0 (protocol crate), Stream C (hex spatial index)
Integrates with: B1 (`update_heightmap_region`), B2 (`mutate_heightmap` + chunk dirtying)

## Goal

Replace the fixed-resolution vertex heightfield with a **terrain operation
log** — analytic geometric operations stacked per hex, evaluated at
arbitrary resolution. A ditch is 40 bytes, not 10,000 modified vertices.
Scales to continent-sized maps and million-edit game histories.

## User stories

**As the simulation engine**, terrain modifications (dig, build, crater,
road) are recorded as compact analytic operations, not dense grids, so
a 100-hour game on a 30km map uses ~40MB of terrain state, not 3.6GB.

**As the wgpu viewer**, terrain mutations arrive as operation deltas,
are rasterized into GPU texture chunks on demand, and render at full
heightmap resolution regardless of how many edits occurred.

**As an entity AI**, I can query terrain height and slope at any world
position in O(ops_in_hex) time, which is bounded by compaction.

**As a player**, I can zoom into a field that was farmed for 50 game-years
and see the accumulated furrows, paths, drainage ditches, and building
foundations — all from the operation history, rendering at full detail.

## Current state

`Heightfield` in `crates/engine/src/v3/spatial.rs`: flat vertex grid at
~75m spacing (2 * map_w + 1 × map_h + 1). `modify_vertex(id, delta)`
mutates individual vertices. `effective_height_at()` does bilinear
interpolation via a `pos_to_vertex` closure. Resolution is tied to hex
geometry — far too coarse for player-scale terrain edits.

Viewer (B1/B2): `HeightmapRenderer` accepts `map_width × map_height`
R32Float texture. `mutate_heightmap(x, y, w, h, data)` updates CPU copy
and marks 64×64 chunks dirty. `flush_dirty_chunks()` uploads to GPU.
Currently initialized with demo procedural data, not engine heightfield.

Protocol: `V3Init` sends `height_map: Vec<f32>` and `material_map: Vec<f32>`
sampled one-per-hex. No incremental terrain messages exist.

## Architecture

```
Terrain(pos) = coarse_base(pos) + Σ ops_in_hex(pos)
```

Three layers:

1. **Coarse base** — existing `Heightfield` at hex-boundary resolution.
   Covers the entire map cheaply. Generated by mapgen. Immutable after
   init (mutations go through operations).

2. **Operation log** — per-hex `Vec<TerrainOp>`. Each op is an analytic
   shape (ditch, wall, crater, road, flatten, furrow). Evaluated
   procedurally at query time — no stored samples.

3. **Rasterization cache** — lazily computed dense patches for viewer
   GPU upload and hot-path physics queries. Invalidated when the hex's
   op stack changes.

### Why operation log, not dense grid

| Concern | Dense 1m grid | Operation log |
|---------|--------------|---------------|
| 3km map, no edits | 36MB | 0 bytes |
| 3km map, 10k edits | 36MB | ~400KB |
| 30km map, 100k edits | 3.6GB | ~4MB |
| 300km map, 1M edits | 360GB | ~40MB |
| Replay/history | Separate delta log | IS the history |
| Resolution | Fixed 1m | Arbitrary (eval at any density) |
| Narrative query | Opaque height values | "What happened here?" |

### TerrainOp enum

```rust
enum TerrainOp {
    /// Linear depression (trench, drainage, irrigation)
    Ditch {
        start: Vec2, end: Vec2,
        width: f32, depth: f32,
        profile: DitchProfile,  // Trapezoidal, V-shaped, U-shaped
    },
    /// Linear ridge (earthen wall, berm, levee)
    Wall {
        start: Vec2, end: Vec2,
        width: f32, height: f32,
        profile: WallProfile,  // Rounded, flat-top, pointed
    },
    /// Radial depression (explosion, sinkhole, quarry)
    Crater {
        center: Vec2, radius: f32, depth: f32,
        rim_height: f32,  // Raised rim from ejecta
    },
    /// Smoothed path between waypoints
    Road {
        points: SmallVec<[Vec2; 4]>,
        width: f32,
        grade: f32,  // Target slope (0 = flat)
        material: GeoMaterial,
    },
    /// Rectangular area leveled to target height
    Flatten {
        center: Vec2, half_extents: Vec2,
        target_height: f32,
        rotation: f32,
    },
    /// Parallel furrows in a bounded region (farming)
    Furrow {
        center: Vec2, half_extents: Vec2,
        rotation: f32,
        spacing: f32, depth: f32,
    },
    /// Cylindrical bore (mine shaft, well, post hole)
    Bore {
        center: Vec2, radius: f32,
        depth: f32, angle: f32,
    },
    /// Compacted snapshot — replaces the entire stack below this point.
    /// Stored as a dense height delta grid for this hex.
    Snapshot {
        origin: Vec2,
        cols: u16, rows: u16,
        cell_size: f32,
        deltas: Vec<f32>,  // Height delta from coarse base
        materials: Option<Vec<GeoMaterial>>,
    },
}
```

The `Snapshot` variant is the compaction target — when a hex's stack
exceeds the threshold, all ops below are rasterized into a Snapshot,
then the old ops are dropped.

### Evaluation

```rust
fn terrain_height_at(pos: Vec2, state: &GameState) -> f32 {
    // 1. Coarse base (bilinear interpolation, same as current)
    let base = state.heightfield.effective_height_at(pos, world_to_vertex);

    // 2. Find containing hex
    let hex = state.hex_mapping.medium_hex(pos);

    // 3. Apply operation stack
    let ops = state.terrain_ops.get(hex);
    let mut delta = 0.0;
    for op in ops {
        delta += op.evaluate_at(pos);
    }

    base + delta
}
```

Each `TerrainOp::evaluate_at(pos)` returns the height delta at that
world position. Operations that don't affect the query point return 0.0
(early-out via bounding box check).

### Compaction

When `ops.len() > COMPACTION_THRESHOLD` (default 50):

1. Rasterize all ops into a dense delta grid at 1m resolution
2. Replace the stack with a single `Snapshot` op
3. New operations stack on top of the snapshot

The snapshot captures accumulated state. History before compaction is
lost from the live state (but preserved in replay if recorded).

Compaction is triggered lazily — on the next op push that exceeds the
threshold. Cost: rasterize ~150×150 = 22.5K samples. At ~50 ops per
sample, ~1.1M evaluations. Should be <1ms for simple analytic ops.

### Op merging (optional optimization)

Before pushing a new op, check if it can merge with the top of stack:
- Two adjacent `Ditch` ops with same width/depth/profile → extend into one
- Two overlapping `Crater` ops → keep both (merging is lossy)
- `Flatten` followed by `Flatten` at same location → replace

Conservative — only merge when the result is identical to sequential
evaluation. This keeps the stack shorter without compaction.

**Evaluator design:**
Each op variant implements:
```rust
fn evaluate_at(&self, pos: Vec2) -> f32;  // height delta
fn bounds(&self) -> Aabb2d;               // early-out bounding box
fn affects(&self, pos: Vec2) -> bool {     // quick rejection
    self.bounds().contains(pos)
}
```

Ditch evaluation example (trapezoidal profile):
- Project pos onto ditch center line → get distance-along and distance-from
- If distance-from > width/2: return 0.0 (outside ditch)
- If within flat bottom: return -depth
- If on slope: interpolate linearly between 0 and -depth

**Tests:**
- Ditch at known position: height at center = base - depth
- Ditch at edge: height transitions smoothly to base
- Wall at known position: height at center = base + wall_height
- Crater: height at center = base - depth, height at rim = base + rim_height
- Stacked ops: ditch + wall across it = correct composite height
- Empty op log: terrain_height_at matches coarse base exactly
- Op outside hex: evaluate_at returns 0.0

**Init rasterization:**
The viewer needs a full heightmap texture at init. Options:
- (a) Rasterize full map server-side at 1m, send as binary blob
- (b) Send coarse base + all ops, viewer rasterizes client-side

Option (a) for now — simpler, one-time cost. A 3km map at 1m = 9M floats
= 36MB. With msgpack + compression, ~8-12MB. Acceptable for init.

For larger maps: switch to (b) or stream chunks on demand (future work).

**Delta flow:**
Each tick, the engine tracks which hexes had terrain ops pushed. For
those hexes, rasterize the patch (from cache) and include in the delta
message. The viewer applies patches to its CPU copy and marks chunks
dirty. `flush_dirty_chunks()` uploads to GPU.

**Tests:**
- Round-trip: TerrainPatch encode → decode produces identical data
- Init rasterization matches point-by-point terrain_height_at
- Delta with terrain patch: viewer height data updated correctly
- Multiple patches in one delta: all applied correctly
- No patches in delta: no terrain changes (no unnecessary uploads)

**Files modified:**
- `crates/engine/src/v3/terrain_ops.rs` — Material-aware evaluation:
  ops can override material (Road → Stone, Crater → charred Soil).
  Add `material_at(pos, state) -> GeoMaterial`.
- `crates/engine/src/v3/sim.rs` — Add terrain action processing phase.
  Entities with active terrain tasks (dig, build, plow) push ops
  per swing cycle.
- `crates/engine/src/v3/state.rs` — Add `TerrainTask` variant to entity
  task system (dig target, build target, plow region).

**Terrain action flow:**
1. Agent AI assigns terrain task to entity (e.g., "dig ditch from A to B")
2. Entity walks to start position
3. Each swing cycle (from kinetic chain, A2): push a small incremental op
   - Digging: `Ditch { depth: 0.1, ... }` per swing (accumulates via
     evaluation — 15 swings = 1.5m deep ditch)
   - Building: `Wall { height: 0.1, ... }` per shovel-load
4. Material hardness affects swing effectiveness:
   - Soil: 1.0× (baseline)
   - Sand: 0.8× (loose, fast to dig, walls crumble)
   - Clay: 1.2× (sticky, slow to dig, walls hold)
   - Rock: 0.2× (very slow, needs pickaxe)

**Incremental ops vs one big op:**
Each swing pushes a tiny op. This is intentional — it means:
- Partial work is visible (half-dug ditch)
- Interruption preserves progress
- Multiple workers on one ditch naturally compose
- Compaction handles the accumulation

**Op merging for incremental digs:**
When pushing an incremental Ditch op, check if top-of-stack is a Ditch
with identical start/end/width/profile. If so, increase depth instead
of pushing a new op. This keeps the stack short for sustained digging.

**Tests:**
- 15 dig swings produce a ditch ~1.5m deep
- Partial dig (5 swings) → ditch is ~0.5m deep
- Two entities digging same ditch from opposite ends → continuous ditch
- Rock terrain: 15 swings produce ~0.3m depth
- Material override: road op changes material_at to Stone
- Interrupted dig preserves partial progress
